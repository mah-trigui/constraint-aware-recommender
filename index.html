Hard Constraints Shouldn’t Be Features — They Should Be the Architecture

A geography-first recommender built for sparse interactions and hard delivery boundaries

View Repository Explore Method

Overview

This project tackles a restaurant recommendation problem with about 100 vendors, sparse user history, and strong delivery-radius constraints.

The central design decision was not to force collaborative filtering onto the problem. Instead, the recommender was built around the dominant marketplace constraint: geography.

Vendor zones were first defined geographically, customers were assigned to the nearest zone, and recommendations were then generated inside each segment × zone pair using historical vendor frequency weighted by customer-vendor proximity.

Architecture

The Core Idea

In many recommendation problems, location is treated as one more input feature. In this project, that would have been the wrong abstraction.

Delivery geography is a hard constraint, not a soft preference. If a vendor is outside the realistic service area, no amount of similarity modeling can make that recommendation useful.

When a signal is a hard constraint, it should shape the architecture before ranking begins.

Why collaborative filtering was not the starting point

Small item space

The catalog contained only around 100 vendors, which reduces the value of large-scale collaborative structure.

Sparse interaction matrix

Most customers had ordered from just one or two vendors, limiting the strength of user-item similarity.

Hard geographic feasibility

Delivery radius is not a weak preference signal. It defines the feasible recommendation set.

Pipeline Design

  • Order history aggregation: build per customer × location × vendor historical summaries
  • Vendor processing: group raw vendor tags into broader cuisine families and clean coordinates
  • Customer profiles: combine demographics, account tenure, and number of saved locations
  • Vendor zones: use K-means on vendor coordinates to create spatial service zones
  • Customer segments: build segments using gender × location count × account tenure
  • Recommendation layer: inside each segment × zone, rank vendors by historical frequency divided by Manhattan distance

3. Assign customers to nearest zone

assign_zone <- function(profiles, centers) {
    zones <- apply(profiles[, .(long, lat)], 1, function(row) {
        dists <- apply(centers, 1, function(ctr) sum((row - ctr)^2))
        which.min(dists)
    })
    zones
}

4. Rank vendors inside each segment × zone pair

dist <- abs(vend_freq$vend_long - cust_long) +
    abs(vend_freq$vend_lat - cust_lat)
dist[dist == 0] <- 0.001

vend_freq$score <- vend_freq$freq / dist
top_vend <- vend_freq[order(-score), vendor_id][1:min(nrow(vend_freq), 10)]

Why this matters

  • shows that recommender-system design starts with marketplace structure, not model defaults
  • separates hard constraints from soft preference signals
  • uses architecture to encode feasibility before ranking
  • generalizes to logistics, delivery, mobility, and location-constrained marketplaces

Where this pattern generalizes

  • food delivery and quick commerce
  • mobility and driver-passenger matching
  • service marketplaces with coverage zones
  • local inventory recommendation under physical reach constraints

Key Takeaway

Before choosing the recommender model, decide which signals are constraints and which are preferences.

If a signal is a hard boundary, it belongs in the architecture rather than being left for the model to discover as just another feature.
